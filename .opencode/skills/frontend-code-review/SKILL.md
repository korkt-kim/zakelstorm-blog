# 프론트엔드 코드 리뷰

## 의도

사용자가 프론트엔드 코드를 검토하도록 요청할 때마다 이 스킬을 사용하세요 (특히 `.tsx`, `.ts` 또는 `.js` 파일). 두 가지 리뷰 모드를 지원합니다:

1. **대기 중인 변경사항 리뷰** – 커밋할 예정인 스테이징된/작업 트리 파일을 검사하고 제출 전에 체크리스트 위반 사항을 표시합니다.
2. **파일 대상 리뷰** – 사용자가 지정한 특정 파일을 검토하고 관련 체크리스트 결과를 보고합니다.

모든 적용 가능한 파일과 모드에 대해 아래 체크리스트를 고수하세요.

## 체크리스트

[references/code-quality.md](references/code-quality.md), [references/performance.md](references/performance.md), [references/business-logic.md](references/business-logic.md)를 참조하세요. 카테고리별로 분류된 현재 체크리스트를 정규 규칙 집합으로 취급하세요.

각 규칙 위반을 긴급도 메타데이터로 표시하여 향후 검토자가 수정 우선순위를 정할 수 있도록 합니다.

## 리뷰 프로세스

1. 관련 컴포넌트/모듈을 엽니다. 클래스 이름, React Flow 훅, props 메모이제이션, 스타일링과 관련된 라인을 수집합니다.
2. 리뷰 포인트의 각 규칙에 대해 코드가 어떻게 벗어났는지 기록하고 대표적인 코드 조각을 캡처합니다.
3. 아래 템플릿에 따라 리뷰 섹션을 작성합니다. 위반 사항을 먼저 **긴급** 플래그로 그룹화한 다음 카테고리 순서(코드 품질, 성능, 비즈니스 로직)로 정렬합니다.


## Output Format [Mandatory]

**CRITICAL INSTRUCTION:**
Review 결과를 출력할 때는 **반드시** 아래 정의된 **Template A** 또는 **Template B** 형식을 그대로 사용해야 합니다.
- 기본 포맷이나 다른 마크다운 형식을 사용하지 마십시오.
- 템플릿의 헤더, 구조, 이모지 등을 변경하지 마십시오

### 템플릿 A (발견된 사항이 있는 경우)

```
# 코드 리뷰
수정해야 할 긴급 문제 <N>개:

## 1 <버그에 대한 간단한 설명>
파일경로: <경로> <라인>
<관련 코드 조각 또는 포인터>


### 제안된 수정
<제안된 수정에 대한 간단한 설명>

---
... (각 긴급 문제에 대해 반복) ...

개선 제안 <M>개:

## 1 <제안에 대한 간단한 설명>
파일경로: <경로> <라인>
<관련 코드 조각 또는 포인터>


### 제안된 수정
<제안된 수정에 대한 간단한 설명>

---

... (각 제안에 대해 반복) ...
```

긴급 문제가 없으면 해당 섹션을 생략합니다. 제안이 없으면 해당 섹션을 생략합니다.

문제 번호가 10개를 초과하면 "10개 이상의 긴급 문제" 또는 "10개 이상의 제안"으로 요약하고 처음 10개의 문제만 출력합니다.

섹션 사이의 빈 라인을 압축하지 마세요. 가독성을 위해 그대로 유지하세요.

템플릿 A를 사용하는 경우 (즉, 수정할 문제가 있는 경우) 그리고 적어도 하나의 문제가 코드 변경을 요구하는 경우, 구조화된 출력 후에 간단한 후속 질문을 추가하여 사용자가 제안된 수정을 적용하기를 원하는지 물어봅시다. 예를 들어: "이러한 문제를 해결하기 위해 제안된 수정 섹션을 사용하시겠습니까?"

### 템플릿 B (문제 없음)

```
## 코드 리뷰
문제를 찾지 못했습니다.
```
